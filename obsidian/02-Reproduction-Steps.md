## BUG-001: Deleting/backspacing reps field deletes entire set or workout

- **Priority**: Critical (Data Loss)

### Root cause analysis

The guard `if (!reps || reps === '0') return;` at **AddSetForm.tsx:26** is NOT the direct cause of data loss. When `reps` is empty string, `!''` is true, so `handleSubmit` returns early with zero side effects. It does NOT trigger deletion.

**The real mechanism is a UI touch-target hazard on mobile:**

1. The **"Del" button** for the most recently logged set lives in the table row at `WorkoutView.tsx:272` directly above the AddSetForm.
2. On narrow mobile viewports, the "Del" button's touch target overlaps or is very close to the AddSetForm's reps `<input>`.
3. When the user reaches to tap the reps input (to backspace and retype), their thumb can accidentally register on the "Del" button.
4. `deleteSet` fires at `App.tsx:185`, removing the targeted set.
5. **Cascade**: If the deleted set was the only set in that exercise and the exercise has no `.notes`, the exercise is filtered out (`.filter(ex => ex.sets.length > 0 || ex.notes)`, line 193).
6. If that was the last exercise, the entire session is nulled via `clearSession()` and `return null` (lines 194-196) — entire workout gone.

### Steps to reproduce (mobile)

1. Start a new workout and add any exercise (e.g., Bench Press)
2. Log at least one set (enter weight and reps, tap "Log")
3. In the AddSetForm directly below the table, tap the reps input to focus it
4. Backspace the pre-filled reps value
5. Instead of tapping "Log", let your thumb land on the area just above the reps input — the last logged set's row where "Del" lives
6. **Actual**: The last set disappears. If only one set existed, exercise may also disappear. If only one exercise existed, UI returns to "Start Workout"
7. **Expected**: Tapping/editing the reps input should have no effect on existing data

### Steps to reproduce (desktop — confirms cascade)

1. Log one set in any exercise
2. Click the **"Del"** button on set #1's row
3. **Actual**: Set deleted. If only set + no notes → exercise removed. If only exercise → session nulled (workout lost)

### Code trace

| Step | File | Line(s) | What happens |
|------|------|---------|-------------|
| User backspaces reps | AddSetForm.tsx | 56 | `setReps('')` — reps state becomes empty string |
| Thumb lands on "Del" | WorkoutView.tsx | 272 | `onClick={() => onDeleteSet(ex.id, set.id)}` fires |
| deleteSet runs | App.tsx | 185-199 | Filters set from exercise's array |
| Empty-exercise cleanup | App.tsx | 193 | `.filter(ex => ex.sets.length > 0 \|\| ex.notes)` — removes exercise |
| Session nulled | App.tsx | 194-196 | `clearSession()` + `return null` — workout lost |

### Why the falsy guard suspicion was wrong

The guard at AddSetForm.tsx:26 is a **silent no-op** when reps is empty. It only prevents sets with `reps: 0` or `reps: NaN` from being added. It is correctly scoped and does not participate in the deletion path.

### Fix recommendations

1. **Increase touch-target separation**: Add bottom margin to the sets table's last row, or move "Del" to the left side of the row
2. **Replace inline "Del" with swipe-to-delete**: Swipe gesture less prone to accidental activation
3. **Add confirmation for cascading deletes**: Before removing last set of an exercise or last exercise of a session, show confirmation
4. **Add undo toast**: After any set deletion, show "Set deleted. Undo?" toast

## BUG-005: Parser uses RegExp lookbehind incompatible with older Safari

- **Priority**: High
- **File**: `src/parser.ts:154`
- **Root cause**: `split(/(?<=\d)\s+and\s+/i)` uses a negative-width lookbehind assertion `(?<=\d)`, which is unsupported in Safari < 16.4 (all iOS 15 devices and some iOS 16). When the regex engine fails to parse the lookbehind, `String.prototype.split()` throws a SyntaxError — crashing the entire voice parser for any compound command containing "and" preceded by a digit.

### Steps to reproduce

1. Use an iPhone running iOS 15.x (or Safari < 16.4 on desktop)
2. Open IronLog, start a workout
3. Use voice input and say "bench press 185 for 10 and squat 225 for 5"
4. **Actual**: Parser crashes with `SyntaxError: Invalid regular expression: invalid group specifier name`. The entire compound voice command is silently dropped (caught by intent router but returns no sets).
5. **Expected**: Voice command splits correctly into "bench press 185 for 10" and "squat 225 for 5", both parsed normally.
6. On modern Safari/Chrome/Firefox: Works correctly (confirms it's a browser-compat issue, not logic bug).

### Code trace

| Step | File | Line(s) | What happens |
|------|------|---------|-------------|
| Voice command arrives | voiceCommands.ts | - | Intent router classifies and passes to rule parser |
| Rule parser tokenizes | parser.ts | 150-158 | `splitOnConjunctions()` called |
| Compound "and" split | parser.ts | 154 | `split(/(?<=\d)\s+and\s+/i)` throws SyntaxError on old Safari |
| Error propagates | parser.ts | - | Parser returns 0 sets, intent silently fails |

### Fix recommendation
Replace lookbehind with capture-group or two-pass approach:
```js
// Option A: capture-based (no lookbehind)
const andParts = part.split(/\s+and\s+/i);
// Rejoin false splits where "and" wasn't preceded by a digit
// Option B: manual loop scanning for /\d\s+and\s+/i positions
```

## BUG-009: writeJSON silences all IDB write failures

- **Priority**: Medium
- **File**: `src/storage.ts:35-39`
- **Root cause**: `void idbSet(key, serialised)` explicitly discards the promise. If IDB write fails (storage quota exceeded, transaction error, iOS kills the transaction), there is zero feedback — no console error, no toast. localStorage remains the sole copy. On iOS, localStorage is evicted after 7 days of inactivity, meaning the user's data survives the session but is silently lost on next app open a week later.

### Steps to reproduce

1. Fill device storage to near-quota
2. Log several workouts in IronLog
3. Close the tab, wait 8+ days on iOS without opening IronLog
4. Reopen IronLog
5. **Actual**: All data is gone. No error was ever shown. The IDB writes all failed silently.
6. **Expected**: If IDB is the durability layer, its failures must be surfaced. At minimum, a failed write should log a warning. Ideally, show a toast: "Storage full — data may not persist after app close."
7. To confirm the silent failure:
   - Open DevTools → Application → IndexedDB
   - Set a breakpoint on `idbSet` or simulate quota with Chrome's Storage quota override
   - Trigger a write, observe the rejected promise is silently swallowed

### Code trace

| Step | File | Line(s) | What happens |
|------|------|---------|-------------|
| Any data write | storage.ts | 35-38 | `writeJSON` writes to localStorage first |
| IDB mirror write | storage.ts | 38 | `void idbSet(key, serialised)` — promise discarded |
| IDB write fails | idb-storage.ts | - | QuotaExceededError, transaction aborted, etc. |

## BUG-010: visibilitySync fires duplicate IDB writes

- **Priority**: Medium
- **File**: `src/idb-storage.ts:84-99`
- **Root cause**: `visibilitySync` iterates every localStorage key on tab-hide and writes all of them to IDB in parallel via `Promise.allSettled`. Since `writeJSON` (in storage.ts:38) already does an `idbSet` on every write, the visibilitychange handler re-writes data that's already in IDB. On large datasets (many workouts), this is a thundering herd of redundant IDB transactions fired as the page is being backgrounded — exactly when iOS is most likely to kill the tab mid-operation.

### Steps to reproduce

1. Open IronLog, accumulate a large history (50+ workouts with many sets)
2. Open DevTools → Application → IndexedDB, watch transaction count
3. Switch to another tab or background the browser
4. **Actual**: Every localStorage key triggers an IDB write, even keys that were already written-through by `writeJSON`. Transaction count spikes on tab-hide.
5. **Expected**: visibilitySync should only sync keys that are newer in localStorage than in IDB (dirty tracking), or the write-through in `writeJSON` should be sufficient and visibilitySync should be a no-op on already-synced keys.

### Code trace

| Step | File | Line(s) | What happens |
|------|------|---------|-------------|
| writeJSON called | storage.ts | 38 | Writes to IDB immediately (write-through) |
| Tab hidden | idb-storage.ts | 88 | visibilitychange fires |
| All keys enumerated | idb-storage.ts | 90-95 | Every localStorage key collected into snapshot |
| Duplicate writes | idb-storage.ts | 97 | `Promise.allSettled(snapshot.map(...))` — re-writes already-persisted data |

### Fix recommendation
- Add a dirty-flag per key: set on `writeJSON`, cleared after visibilitySync successfully flushes. Skip clean keys.

But if both numbers are under 30 (e.g., "15 15" for 15lb x 15 reps), the heuristic can't disambiguate — it defaults to first-as-weight, second-as-reps, which is a coin flip.

### Steps to reproduce

5. Test edge: "lateral raise 10 25" (10lb x 25 reps)
6. **Actual**: `10 <= 30` and `25 < 30` → else branch → `weight: 10, reps: 25` — correct by luck
7. Test edge: "lateral raise 25 10" (25lb x 10 reps)
8. **Actual**: `25 <= 30` and `10 < 30` → else branch → `weight: 25, reps: 10` — correct by luck
9. The real failure: "lateral raise 5 45" (5lb x 45 reps)
10. **Actual**: `5 <= 30` and `45 >= 30` → second branch fires → `weight: 45, reps: 5` — **swapped!** 45lb lateral raise logged instead of 5lb.

### Code trace

## BUG-016: shareWorkoutAsFit downloads file on cancelled share (some platforms)

- **Priority**: Low
- **File**: `src/share.ts:108-117`
- **Root cause**: The Web Share API's promise rejection on cancel is supposed to throw `AbortError`, which line 112 handles. But some browsers/platforms throw a different error type on cancel (e.g., `DataError` on Firefox Android, generic `Error` on some WebView implementations). When the error name isn't `AbortError`, the catch block falls through to the download fallback (line 115), forcing a file download the user explicitly cancelled.

### Steps to reproduce

1. Open IronLog on Firefox Android (or a device with a non-standard share cancel behavior)
2. Go to History → tap a workout → tap Share → choose "FIT file"
3. When the system share sheet appears, tap Cancel
4. **Actual**: A FIT file downloads to the device. The user cancelled the share but got a download anyway.
5. **Expected**: Cancelling the share sheet should do nothing — no download, no side effects.
6. On Chrome/Safari with proper AbortError: Cancel works correctly (confirms platform-specific issue).

### Code trace

| Step | File | Line(s) | What happens |
|------|------|---------|-------------|
| Share invoked | share.ts | 108 | `share({ files: [chosen] })` called |
| User cancels | share.ts | 110 | Promise rejects |
| Error check | share.ts | 111-112 | `err.name === 'AbortError'` — matches on Chrome/Safari |
| Fallthrough | share.ts | 115 | Unknown error name → `triggerDownload(bytes, filename)` fires |

### Fix recommendation
Treat any non-technical rejection (not a real network/permission error) as cancellation:
```js
.catch((err) => {
  // Treat all share rejections as user cancellation — the API spec is inconsistent
  if (err instanceof Error && err.name === 'AbortError') return { result: 'cancelled', ... };
  // Fallback: only download if it looks like a real error (e.g., files not supported)
  if (err instanceof Error && err.name === 'DataError') return { result: 'cancelled', ... };
  triggerDownload(bytes, filename);
  return { result: 'downloaded', ... };
});
```

## BUG-017: debugLogger stack trace captured at log-time, not error-time

- **Priority**: Low
- **File**: `src/debugLogger.ts:99`
- **Root cause**: `new Error().stack` at line 99 captures the stack at the call to `push()`, not at the original error site. If an error is caught in a try/catch, wrapped, and re-logged, the stack trace points to the logger's `push()` function rather than the original throw site. This makes debugging harder — the stack shows the logging infrastructure, not the bug.

### Steps to reproduce

1. Trigger an error path in the app (e.g., fail a fetch, throw in a handler)
2. The error is caught and passed to `debugLogger.error(err)`
3. Open the debug log viewer
4. **Actual**: Stack trace for the entry starts at `push()` (debugLogger.ts:99), not at the original error origin
5. **Expected**: Stack trace shows where the error was originally thrown/caught

### Code trace

| Step | File | Line(s) | What happens |
|------|------|---------|-------------|
| Error thrown | any file | - | Error object created with original stack |
| Caught and logged | debugLogger.ts | 93 | `push('error', [err])` called |
| New stack created | debugLogger.ts | 99 | `new Error().stack` — captures HERE, not original site |
| Original stack discarded | debugLogger.ts | 98-100 | `entry.args` has the error but `entry.stack` is wrong |

### Fix recommendation
When logging an error-level entry, check if any of the `args` is an `Error` instance and use its `.stack` instead of creating a new Error:
```js
stack: level === 'error'
  ? (args.find(a => a instanceof Error) as Error)?.stack?.split('\n').slice(2).join('\n')
    ?? new Error().stack?.split('\n').slice(2).join('\n')
  : undefined,
```

## BUG-018: Parser compound AMRAP/drop-set detection order issue

- **Priority**: Low
- **File**: `src/parser.ts` — various locations
- **Root cause**: The rule parser processes patterns in a fixed order — warmup → drop set → AMRAP (line 289 checks `dropSets.length === 0`) → RPE → multi-set → bare number pairs. If a voice command contains overlapping patterns (e.g., "185 for 10 then drop set 155 to 135 AMRAP"), the drop-set prefix regex at line 270 and AMRAP regex at line 290 may compete for the same text. The order-dependent parsing can produce incorrect set arrays for complex compound commands.

### Steps to reproduce

1. Start a workout, add any exercise
2. Use voice input: "bench press 185 for 10 then drop set 155 to 135 AMRAP"
3. **Actual**: The drop set prefix matches at line 270, consuming "drop set 155". But "to 135 AMRAP" — the "AMRAP" suffix on the drop chain is not recognized because AMRAP parsing at line 289 is guarded by `dropSets.length === 0`. Result: AMRAP ignored, the last drop set is logged as a normal set.
4. **Expected**: Drop set chain with the final drop being AMRAP: 185x10 (normal), 155x? (drop), 135xAMRAP (drop + AMRAP).
5. Try: "135 AMRAP then drop set 115" — AMRAP matches first, drop set hits the `dropSets.length === 0` guard on AMRAP, but AMRAP was already parsed. Works in this order but fails in reverse.

### Code trace

| Step | File | Line(s) | What happens |
|------|------|---------|-------------|
| Drop set detected | parser.ts | 270-285 | `dropSets` populated, `remaining` advanced |
| AMRAP check skipped | parser.ts | 289 | `if (dropSets.length === 0)` — false, AMRAP parsing skipped entirely |
| AMRAP info lost | parser.ts | 287-298 | AMRAP pattern in remaining text silently ignored |

### Fix recommendation
- Remove the mutual exclusion between drop sets and AMRAP: parse AMRAP on the remaining text after drop-set extraction regardless of whether drop sets were found
- Apply AMRAP flag to the last drop set in the chain if the remaining text after drop parsing contains an AMRAP marker

## BUG-019: No error boundary in React tree

- **Priority**: Low
- **File**: `src/main.tsx:6-10`
- **Root cause**: The React tree at `main.tsx:9-12` renders `<App />` inside `<React.StrictMode>` with no error boundary. Any uncaught exception during render (e.g., accessing a property on undefined after corrupted localStorage data, a third-party library throwing) crashes the entire PWA to a blank white screen. Since this is a PWA, the user has no browser chrome to refresh from.

### Steps to reproduce

1. Open DevTools → Application → localStorage
2. Manually corrupt a key (e.g., set `il-history` to `{invalid json`)
3. Refresh the app
4. **Actual**: White screen. The uncaught `JSON.parse` error crashes the entire React tree. User must manually clear site data or reinstall the PWA.
5. **Expected**: Error boundary catches the crash, shows a fallback UI: "Something went wrong. [Reload] [Reset Data]"
6. Alternative reproduction: Add a throw in any render function, refresh.

### Code trace

| Step | File | Line(s) | What happens |
|------|------|---------|-------------|
| App renders | main.tsx | 10-12 | `<App />` mounts inside StrictMode |
| Error thrown in render | any component | - | Uncaught exception propagates |
| No boundary catches | main.tsx | 9-12 | React unmounts entire tree |
| White screen | - | - | PWA is dead until manual intervention |

### Fix recommendation
Wrap `<App />` in an error boundary component:
```tsx
<React.StrictMode>
  <ErrorBoundary fallback={<ErrorFallback />}>
    <App />
  </ErrorBoundary>
</React.StrictMode>
```
Error boundary should offer "Reload" and "Reset All Data" buttons, and log the error stack via `debugLogger`.
